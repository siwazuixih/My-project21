# Codex Unity Development Instructions

你是我的 Unity 项目开发助手。这个项目主要使用 Unity + C#，目标是稳定、可运行、方便我继续开发。你每次开始任务前、修改代码前、结束任务前，都必须遵守下面规则。
每次收到任何任务时，都必须自动读取 `.codex/WORKLOG.md` 和 `.codex/PROJECT_CONTEXT.md`，不需要用户重复提醒。

## 0. 每次开始任务前必须先做的事

在做任何修改前，先读取并理解这些文件：

1. `AGENTS.md`：当前项目规则。
2. `.codex/WORKLOG.md`：历史开发记录、已完成内容、未完成问题。
3. `.codex/PROJECT_CONTEXT.md`：项目结构、重要脚本、场景、对象命名、机械臂/控制逻辑等长期上下文。
4. 如任务涉及具体脚本，还要先读取相关 `.cs` 文件、场景说明、Prefab 或配置文件。

如果这些文件不存在，先创建它们。不要因为聊天上下文里没有提到就假设项目是空的。

## 1. 记忆与记录规则

每次完成一个阶段后，都要更新 `.codex/WORKLOG.md`，记录内容必须包括：

* 日期和本次任务目标。
* 读取了哪些关键文件。
* 修改了哪些文件。
* 具体做了什么改动。
* 为什么这样改。
* 当前是否完成。
* 还存在什么问题。
* 下次继续开发时应该从哪里开始。
* 需要我手动在 Unity Editor 里检查什么。

如果发现项目结构、核心脚本职责、重要 GameObject 名称、通信协议、串口设置、坐标系约定、机械臂控制逻辑等长期信息发生变化，要同步更新 `.codex/PROJECT_CONTEXT.md`。

不要把密钥、账号、密码、Token、隐私信息写进记录文件。

## 2. Unity 项目开发原则

修改 Unity 项目时要非常谨慎：

* 优先修改 `Assets/` 下的脚本和资源。
* 不要随意删除或大改 `.meta` 文件。
* 不要随意修改 `ProjectSettings/`，除非任务明确需要。
* 不要修改 `Library/`、`Temp/`、`Obj/`、`Build/`、`Logs/`、`.vs/` 等自动生成目录。
* 修改场景 `.unity`、Prefab `.prefab` 文件前要先说明风险，因为这些 YAML 文件容易产生难以 review 的大改动。
* 能用脚本解决的，不要优先直接改场景文件。
* 保持 C# 代码清晰、可读、少耦合。
* 不要为了“看起来完成”而写伪代码或空实现。

## 3. C# 代码规范

写 Unity C# 脚本时遵守：

* 类名、文件名保持一致。
* 公共字段尽量使用 `[SerializeField] private`，除非确实需要 public API。
* 在 `Update()` 中避免高成本操作。
* 不要在 `Update()` 中频繁 `FindObjectOfType`、`GameObject.Find`、`GetComponent`，应缓存引用。
* 和 Rigidbody、物理相关的逻辑优先放在 `FixedUpdate()`。
* 注意 Unity 主线程限制，不要在后台线程里直接操作 Unity API。
* 对串口、网络、机器人通信、文件 IO 等外部资源要有异常处理和释放逻辑。
* 日志使用 `Debug.Log`、`Debug.LogWarning`、`Debug.LogError`，但不要刷屏。
* 不要破坏已有 public 方法名，除非同步修改所有引用。

## 4. 机械臂 / 真实机器人 / 仿真控制相关规则

如果任务涉及机械臂、Dobot、末端执行器、升降气缸、真实机器人跟随、坐标同步等内容：

* 先梳理当前控制链路：输入来源、坐标转换、目标对象、控制脚本、执行对象。
* 区分“Unity 仿真对象”和“真实机器人控制对象”。
* 不要在没有确认安全逻辑的情况下直接让真实机器人执行危险动作。
* 对真实设备相关代码保留安全开关，例如 `enableRealRobot`、`dryRun`、`simulationOnly`。
* 对移动范围、速度、加速度、关节限制、末端位置要做边界检查。
* 坐标系转换要写清楚来源和单位，例如 Unity 米制坐标、机器人毫米坐标、世界坐标、本地坐标。
* 如果不确定硬件协议，不要猜测发送危险命令，先写成可配置或模拟模式。

## 5. 工作流程

每次接到任务后，按这个流程执行：

1. 先阅读 `AGENTS.md`、`.codex/WORKLOG.md`、`.codex/PROJECT_CONTEXT.md`。
2. 用 `git status` 查看当前工作区状态。
3. 如果已有未提交修改，先说明哪些文件已有改动，避免覆盖我的工作。
4. 分析任务，列出简短计划。
5. 只修改和任务相关的最小范围文件。
6. 修改后检查语法、引用、命名空间、Unity 生命周期方法。
7. 能运行测试或编译检查就运行；不能运行时说明原因。
8. 用 `git diff` 总结改动。
9. 更新 `.codex/WORKLOG.md`。
10. 最后给我一个简洁总结：改了什么、如何验证、下一步建议。

## 6. 完成标准

一个任务只有在满足这些条件后才算完成：

* 相关代码已经实际修改完成。
* 没有明显语法错误。
* 没有无意义的大范围重构。
* 没有破坏已有接口和场景引用。
* 已更新 `.codex/WORKLOG.md`。
* 如果产生新的长期项目知识，已更新 `.codex/PROJECT_CONTEXT.md`。
* 已告诉我在 Unity Editor 中需要检查的内容。

## 7. 和我沟通的方式

请用中文回复我。
说明问题时尽量具体，不要只说“已修复”。
如果有多个方案，先推荐最稳妥的方案。
如果不确定 Unity 场景里的对象绑定情况，要明确告诉我需要我在 Inspector 里检查哪些字段。
不要一次性做太多无关改动，优先小步修改、方便回退。
